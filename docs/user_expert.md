# 用户专家（user_expert）· 审查视角档案

> 本文件沉淀"用户专家"审查视角的方法论，用于在 README、SKILL.md、用户输出、教学文档等"用户面"内容上线前做一次"普通用户能否无障碍使用"的把关。
>
> 与 [README §"15 份专家人设"](../README.md#-15-份专家人设招牌功能) 中的 9 active 投资专家不同——这些是"决定买不买"的视角；本视角是"用户愿不愿用、会不会用、用了爽不爽"的视角。
>
> 状态：active（v2.2.0 起沉淀为独立审查方法论）
>
> 数据单位：审查结果用 ✅/⚠️/❌ 三态，耗时以秒/分钟计。

---

## 一、核心哲学

用户专家的本质是**让"完全不懂这套工具的用户"在 30 秒内决定要不要继续、在 5 分钟内首次跑通、在 30 分钟内形成习惯**。它关心的不是分析对不对，而是"用户能否完整走完一次有意义的操作"。三条底层假设：

1. **用户没有耐心**：3 秒看不到想要的东西就会跳走；5 分钟没跑出第一个结果就会放弃。
2. **用户没有背景**：他们不知道"5 层框架""专家圆桌""PE/ROE"的含义，所有术语首次出现都要给出"一句话翻译"或可点击的科普入口。
3. **用户容易自责**：报错信息太技术化会让用户觉得"我哪里搞错了"，实际上 80% 的场景是工具本身设计粗糙。
4. **用户分层而非统一**：[persona.md §"用户优先级"](./persona.md#用户优先级) 三类——散户（60%）/ 学习者（30%）/ 量化（10%），同一段文字要让最大占比的那层 1 眼能看懂，另外两层有"展开看"的退路。

## 二、审查清单（可量化）

下表是用户专家视角的 12 项硬指标，**任何一项 ❌ 即视为 user-P0 缺陷**，必须修复后再上线用户面文档/CLI。

| 维度        | 检查项                                                                                    | 阈值                | 状态 |
| ----------- | ----------------------------------------------------------------------------------------- | ------------------- | ---- |
| 首屏可懂    | 进入首页/Hero 段后 5 秒内能否理解"这是什么 + 怎么开始"                                    | 一句话 + 一个命令   | ✅/⚠️/❌ |
| 5 分钟跑通  | 全新用户从"克隆项目"到"看到第一个有意义的输出"能否在 300s 内完成                          | ≤ 5 min             | ✅/⚠️/❌ |
| 默认值完备  | 关键配置（数据源/股票池/通知通道）首次跑是否能"零配置可用"，无 token 不报错               | 100% 覆盖           | ✅/⚠️/❌ |
| 错误友好    | 数据源挂、网络断、股票代码错时的报错是否给出"下一步该做什么"而非堆栈                      | 必须给具体动作      | ✅/⚠️/❌ |
| 命令别名    | 同一意图是否有 ≥ 1 个常用别名（"调仓建议"=rebalance / "持仓"=portfolio / "选股"=screener）| 必含                | ✅/⚠️/❌ |
| 自然语言    | `portfolio-natural` / 用户口语指令（"我买了 1000 股宝丰能源"）能否被正确映射到 API         | 3 条以上典型句式    | ✅/⚠️/❌ |
| 输出稳定    | 同一只股票同一参数连续跑 3 次，输出结构是否 100% 一致                                      | 必达                | ✅/⚠️/❌ |
| 术语解释    | PE/ROE/PEG/缠论/胜率这类术语首次出现是否有"一句话翻译"或 `methodology.md` 跳链            | 必含                | ✅/⚠️/❌ |
| 学习曲线    | 从 quick → full → debate → technical，跨度是否平滑（每升一级只增加 1 个新概念）           | 每升 +1 概念        | ✅/⚠️/❌ |
| 卸载路径    | 用户觉得"不好用"时，能否在 1 条命令内回到出厂状态（`rm` 软链 + 清缓存 + 配置复位）       | 1 条命令            | ✅/⚠️/❌ |
| 新手友好    | `/help`（= `/stock-help`）能否解决"我现在该用啥" 80% 的提问                              | ≥ 80% 一问即答      | ✅/⚠️/❌ |
| 风险提示    | 任何含"买/卖/止损/仓位"结论的输出是否在最显眼位置出现"仅供参考，不构成投资建议"           | 必含                | ✅/⚠️/❌ |

> **联动工具**：`scripts/dev/check_allowed_tools.py`（自审计 + CI）+ `skills/stock/reports/full-template.md`（首行结论 / 尾行数据源的统一格式）共同保证 user-P0 在工程层面落地。

## 三、优化原则（可执行）

对照清单 ✅ 后，按以下原则做"加分项"，把用户从"能用"推到"爱用"。

### 3.1 首屏三件套（≤ 5 行）

任何用户面文档首屏（Hero）必须包含且只包含三件事：

1. **一句话定位**（不超过 30 字）：例如 *"装进 Claude Code，对话框打 `/stock 贵州茅台`，3 分钟拿到 5 层分析。"*
2. **一个最短路径**（不超过 3 步）：通常是 `git clone + install + 一个演示命令`。
3. **一个"适不适合我"判定**（用表格给 ✅ / ❌ 两列）：让 30% 不匹配的用户 3 秒跳走，避免后期挫败感。

> 反例：首屏塞 5 张架构图 + 6 个链接 + 3 个徽章——视觉密度高但用户抓不到入口。

### 3.2 5 分钟上手（"复制可跑"原则）

`/screener init default` → `/stock sh600519 quick` 是项目的"5 分钟门面"。要求：

- 每一步的命令都是**完整可复制粘贴**（不含省略号、占位符）。
- 输出示例必须给一段**真实运行结果**（不是 LLM 编的占位），让用户对照自己跑出来的结果。
- 三步跑完之后给一个"下一步"引导（通常指向 `/stock sh600519 full` 或 `/portfolio`）。

### 3.3 错误信息的"三明治"结构

数据源挂、网络断、代码错时的报错必须按 [症状 / 根因 / 下一步] 三段写：

```text
❌ 无法获取 sh600519 的财务数据

【症状】所有 fetcher 都返回 5xx / 超时
【根因】30% 是交易所限频，70% 是本地网络；具体见 `data/cache.py` 的 circuit_breaker 状态
【下一步】① 等 60s 再试；② 改用 `/screener init default` 走预置数据；③ 联系 issue 上报
```

绝对不要单独堆栈，也不允许"出现 ERRO 0xFFFF，请重试"这类无意义代号。

### 3.4 输出模板的"首行 + 尾行"约束

所有 skill 的输出统一两件事（[methodology.md §四 输出模板](../methodology.md#四统一输出模板-v18) 详述）：

- **首行必须是结论**（"可分批介入"/"建议减仓 30%"），不是"以下是分析结果"。
- **尾行必须标注数据源 + 时间戳**（`📊 数据源：东财 + 雪球 | 时间：2026-07-01 15:30`），保证可复盘。

这条原则的核心目的：**让用户每次收到输出时，不需要从头读就知道"结论是什么 + 数据来自哪里"**。

### 3.5 学习曲线"扁平化"

从 quick 到 debate，**每升一级只允许新增 1 个概念**：

| 模式       | 新增概念                                       | 用户预期       |
| ---------- | ---------------------------------------------- | -------------- |
| `quick`    | 基本面/估值/技术 三层                           | 3 分钟看结论   |
| `full`     | + 板块层 + 风险收益比                          | 5 分钟看全图   |
| `debate`   | + 专家圆桌 + 投票机制                          | 10 分钟看决策 |
| `technical`| + 均线/MACD/KDJ/缠论/战法（独立技术体系）     | 10 分钟看买点 |
| `research` | + DCF/敏感性分析/排雷                          | 30 分钟看深度 |

跨级跳（如直接进 debate）必须给"如果你不熟悉 X，先看 Y"的引导。

## 四、边界与陷阱

### 4.1 用户专家**不**做的事

- ❌ 不重写前端（[implementation-plan §"不做清单"](./implementation-plan-2026-q3-q4.md#不做清单防止范围蔓延) 第 4 条）
- ❌ 不替投资专家下结论（只调整呈现方式，不动打分逻辑）
- ❌ 不擅自添加商业模式 / 用户画像（只反映 [persona.md](./persona.md) 已定义分层）
- ❌ 不为了"友好"而引入额外三方依赖（项目硬约束"零运行时三方"）

### 4.2 用户专家**最容易踩**的坑

| 陷阱                          | 表现                                                          | 缓解                                                                |
| ----------------------------- | ------------------------------------------------------------- | ------------------------------------------------------------------- |
| **过度抽象**                  | 用"信号""势能""结构"等抽象词替代"涨了多少/跌了多少"          | 数字优先、抽象解读次之；任何抽象词必须配一句"翻译"                  |
| **选项堆叠**                  | 6 种策略 × 9 因子维度 → 命令一次塞 18 个参数                   | 拆成"策略 + 可选覆盖"两层；高频参数写 alias（`--top`/`--all`）     |
| **错误归因**                  | 把技术栈 bug 报成"用户操作错误"                                | 错误信息三明治：症状 / 根因 / 下一步；并标注"这是工具问题不是你的问题"|
| **国际化偏差**                | 默认隐含中文/沪市，深市/港股用户被绕开                        | 命令同时支持 `sh/sz/纯数字/中文名`；错误信息按用户输入回显 code    |
| **首次跑挂**                  | 用户卡在第一步（克隆/装包/股票池初始化）就放弃                 | 提供 `init_pool.py --default` 离线模式作为兜底                       |
| **风险提示弱化**              | 结论过度自信、缺免责声明                                      | 任何含"买/卖/止损/仓位"的输出，最末行强制提示 + 显眼⭐标注          |

## 五、A 股用户特征（参考）

写作任何用户面文案前，先把这三类用户的"动作 + 痛点"背熟：

| 画像       | 占比 | 关键动作                       | 关键痛点                           | 文案语气                     |
| ---------- | ---- | ------------------------------ | ---------------------------------- | ---------------------------- |
| 散户投资者 | 60%  | 看大盘、看持仓、看一眼决策     | 信息过载、技术障碍、时间有限       | 短句、结论先行、风险显眼     |
| 投资学习者 | 30%  | 读五层分析、看 8 位专家怎么想  | 不懂术语、缺乏反馈、无处练手       | 教程化、术语解释、模拟盘引导 |
| 量化爱好者 | 10%  | 调 API、改因子、批量跑回测     | 缺文档、扩展难、性能瓶颈           | JSON / -j / 代码块 / API 链接 |

详细画像与设计启示见 [docs/persona.md](./persona.md)；三类的具体需求映射见 [docs/implementation-plan-2026-q3-q4.md](./implementation-plan-2026-q3-q4.md)（Sprint 7-8 用户研究段）。
