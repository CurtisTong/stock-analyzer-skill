# ADR-002: 熔断器模式（Circuit Breaker）

| 字段     | 值                              |
| -------- | ------------------------------- |
| 状态     | Accepted                        |
| 日期     | 2026-03-20                      |
| 决策者   | curtis                          |
| 相关模块 | `scripts/common/circuit_breaker.py`, `scripts/common/fetcher_base.py` |

## 背景

系统对接 27 个数据源（腾讯/东财/新浪/雪球/THS/akshare 等），
任一数据源可能因限速（429）、宕机、DNS 故障而不可用。
若每次请求都尝试已故障的数据源，会导致响应延迟堆积。

## 决策

采用**三态熔断器**模式保护每个数据源：

```
closed ──(连续失败 ≥ threshold)──> open
open  ──(冷却时间过后)──> half-open
half-open ──(探测成功)──> closed
half-open ──(探测失败)──> open
```

- `closed`（正常）：请求正常通过
- `open`（断开）：直接拒绝请求，不发起网络调用
- `half-open`（半开）：放行少量探测请求，成功则恢复

**关键设计**：
- 429 限速 **不触发** 熔断（限速通常是 API key 级别，其他源未必受限）
- 4xx 业务错误 **不触发** 熔断（如 404 数据不存在，非数据源故障）
- 仅网络异常/5xx 触发 `on_failure()`

## 理由

1. **故障隔离**：坏数据源不影响好数据源，`DataFetcherManager` 按优先级自动切换
2. **自愈能力**：half-open 探测让临时故障恢复后自动重新接入
3. **精准触发**：区分限速/业务错误/网络故障，避免误熔断

## 后果

- 正面：系统在部分数据源故障时仍能降级服务
- 正面：`_last_error` 暴露故障信息供诊断
- 负面：熔断状态为进程内（非分布式），多进程部署时各进程独立熔断
