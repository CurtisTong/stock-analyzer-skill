# 用户画像（Persona）

> 基于项目定位、README 用户分层和三方审查反馈，定义 3 类核心用户画像。

---

## 画像 1：散户投资者（主要用户）

### 基本信息

| 维度         | 描述                    |
| ------------ | ----------------------- |
| **年龄**     | 25-45 岁                |
| **投资经验** | 1-5 年，有 A 股交易经验 |
| **持仓规模** | 5-50 万元               |
| **持仓数量** | 3-10 只                 |
| **技术能力** | 会用终端，但不写代码    |
| **投资风格** | 混合（长线 + 短线）     |

### 核心需求

1. **快速判断**：今天市场怎么样？我的持仓安全吗？
2. **选股辅助**：有什么好的候选？帮我筛一下
3. **风险预警**：哪些持仓需要处理？什么时候该卖？
4. **操作建议**：买多少？止损设在哪？

### 使用场景

- **每日开盘前**：`/market quick` 看大盘，`/portfolio health` 看持仓
- **盘中异动**：`/monitor scan` 检查触发条件
- **周末复盘**：`/stock <持仓股> debate` 深度分析
- **发现机会**：`/screener` 筛选候选，`/stock <候选> quick` 快速验证

### 痛点

- 信息过载：看不过来，不知道哪个重要
- 决策犹豫：买还是不买？卖还是不卖？
- 技术障碍：命令太多记不住，参数不知道怎么填
- 时间有限：每天只有 5-10 分钟看盘

### 对产品的期望

- 一句话结论，不要长篇大论
- 操作建议明确（买入/持有/回避 + 仓位）
- 输出稳定（每次跑结果结构一致）
- 新手友好（`/help` 能解决 80% 问题）

## 画像 2：投资学习者（次要用户）

### 基本信息

| 维度         | 描述                     |
| ------------ | ------------------------ |
| **年龄**     | 20-35 岁                 |
| **投资经验** | 0-2 年，正在学习         |
| **持仓规模** | 1-10 万元（或模拟盘）    |
| **技术能力** | 会用 Claude Code，愿意学 |
| **投资风格** | 探索中                   |

### 核心需求

1. **理解框架**：五层分析怎么看？专家圆桌怎么解读？
2. **学习选股**：什么指标重要？怎么判断好坏？
3. **验证想法**：我觉得这只股票好，帮我分析一下
4. **积累经验**：用模拟盘练习，不亏真金白银

### 使用场景

- **学习模式**：`/stock <股票> full` 看完整五层分析，对照 methodology.md 理解
- **专家解读**：`/stock <股票> debate` 看 8 位专家怎么想，学习多空博弈
- **模拟练习**：`/portfolio web --virtual` 用虚拟盘练手
- **策略验证**：`/backtest` 回测策略，看历史表现

### 痛点

- 不懂术语：PE、PEG、ROE、缠论是什么？
- 不知道看什么：五层分析信息太多，抓不住重点
- 缺乏信心：不知道自己的判断对不对
- 没有反馈：买了之后不知道对错

### 对产品的期望

- 术语解释（PE 是什么？为什么重要？）
- 教程引导（`/help` → 场景 → 步骤）
- 专家解读（8 位专家的人设和逻辑可理解）
- 模拟盘（先练手，再实战）

## 画像 3：量化爱好者 / 工程师（小众用户）

### 基本信息

| 维度         | 描述                    |
| ------------ | ----------------------- |
| **年龄**     | 25-40 岁                |
| **投资经验** | 3-10 年，有编程能力     |
| **持仓规模** | 10-200 万元             |
| **技术能力** | Python 开发者，熟悉 CLI |
| **投资风格** | 系统化 / 量化           |

### 核心需求

1. **数据获取**：行情、财务、K 线数据的 API
2. **策略回测**：多因子策略的历史表现
3. **二次开发**：集成到自己的工作流
4. **参数调优**：策略权重、阈值的优化

### 使用场景

- **数据采集**：`python3 scripts/quote.py sh600519 -j` 获取 JSON 数据
- **策略开发**：修改 `scripts/strategies/` 中的权重和因子
- **批量筛选**：`/screener --full-market --strategy balanced --top 20`
- **回测验证**：`/backtest --all` 对比 5 种策略

### 痛点

- 缺少文档：不知道 fetcher 的 API 格式
- 扩展困难：想加新数据源但不知道怎么写
- 性能瓶颈：全市场筛选太慢
- 缺少指标：想要更多技术指标 / 因子

### 对产品的期望

- API 文档（fetcher 接口、数据类型、返回格式）
- 插件机制（自定义 fetcher / 策略）
- JSON 输出（所有脚本支持 `-j`）
- 代码质量（类型注解、测试覆盖、文档齐全）

## 设计启示

### 针对散户投资者

- 输出模板统一（首行结论 + 尾行数据源）
- `/help` 场景化引导
- `portfolio_web` 默认开浏览器
- 自然语言兼容（"买入了 1000 股宝丰能源"）

### 针对投资学习者

- walkthrough 教程（12 skill 完整流程）
- 专家人设可理解（每位专家有核心哲学、选股标准、代表案例）
- 模拟盘（`--virtual` 模式）
- 术语解释（methodology.md）

### 针对量化爱好者

- JSON 输出（`-j` 参数）
- API 文档（docs/api-reference.md）
- 代码质量（mypy strict、测试覆盖）
- 可扩展架构（BaseFetcher + DataFetcherManager）

## 与其他审查视角的串联

本文件定义的是"用户是谁、要什么"的画像，回答 product 端问题。在三类画像的诉求转换为设计决策时，会引入另外两份审查视角档案来把关：

| 视角         | 角色            | 回答什么                              | 输出锚点                                       |
| ------------ | --------------- | ------------------------------------- | ---------------------------------------------- |
| **用户专家** | `user_expert`   | 用户愿不愿用、会不会用、用了爽不爽    | [docs/user_expert.md](./user_expert.md)       |
| **视觉专家** | `visual_expert` | 信息密度、视觉层级、扫描路径舒不舒服  | [docs/visual_expert.md](./visual_expert.md)   |
| **本档案**   | `persona`       | 用户是谁、各自要什么、分别怎么服务    | [docs/persona.md](./persona.md)（本文件）     |

> **串联使用流程**：本档案定义画像 + 诉求 → `user_expert.md` 12 项清单把关"5 分钟路径 / 错误友好 / 学习曲线" → `visual_expert.md` 14 项清单把关"信息密度 / 表格或列表 / 终端排版" → 进工程实现。

### 把画像映射到审查清单（示例）

| 画像       | 占比 | user_expert 关键指标                  | visual_expert 关键指标         |
| ---------- | ---- | ------------------------------------- | ------------------------------ |
| 散户投资者 | 60%  | 5 分钟上手、错误友好、风险提示显眼    | 首屏三件套、表格胜于列表       |
| 投资学习者 | 30%  | 术语解释、教程引导、模拟盘入口        | mermaid 流程图、详情折叠       |
| 量化爱好者 | 10%  | API 文档、JSON 输出、扩展性           | 代码块完整性、链接文案        |
