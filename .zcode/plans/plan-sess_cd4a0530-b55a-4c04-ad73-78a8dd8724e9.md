# 其余模块深度审查与优化实施计划

## 背景与已确认范围

专家系统审查已完成（2514 测试通过）。本计划处理**其余模块**（数据/抓取器层、策略/技术分析层、业务逻辑层）的审查发现。

已通过直接阅读源码验证 **6 个 P0** + **3 个高影响 P1**。关键事实：
- **P0-1 双重归一化**：3 个 quote fetcher（tencent/sina/eastmoney）在 `fetch()` 内调用 `normalize_volume/normalize_amount`，随后 `scripts/data/__init__.py:230-231` 的 `_dict_to_quote` 又调用 `_normalize_volume/_normalize_amount`（纯粹委托 common.normalize）**第二次**归一化。tencent volume 被 ×100 两次（实际 ×10000），amount ×10000 两次（实际 ×1e8）。**缓存也被污染**（`__init__.py:106` 缓存的是双重归一化后的 `quote.to_dict()`）。K线路径不受影响（kline fetcher 不调用 normalize，`_dict_to_kline_bar:296` 是唯一归一化点）。
- **P0-3 校准数据**：`data/expert_calibration.json` 的 `experts` 键是 legacy 名（buffett/duan_yongping/chaogu_yangjia/zuoshou_xinyi 等），但 `record_prediction` 现在写入 merged active 名（value_institution/emotion_tech）。`verify_predictions:232-233` 用 `if expert_name in data["experts"]` 过滤 → **新预测的专家校准数据永远不更新**（静默数据丢失）。
- **P0-5 中枢合并**：`chan_zhongshu` 初始构建（zhongshu.py:25-26）正确用交集 `zg=min, zd=max`；但合并步骤（:51-57）错误用并集 `max(zg)/min(zd)`，扩大中枢范围，违反缠论定义。现有测试 `test_merge_overlapping_zhongshu` 仅断言 `<=3`，修复安全。

## 阶段 A：P0 正确性修复（6 项）

### A1. Quote 双重归一化修复（P0-1）
**文件**: `scripts/data/__init__.py`、`scripts/fetchers/quote/tencent_quote.py`、`sina_quote.py`、`eastmoney_quote.py`

**根因**: 归一化责任分散在两层。`_dict_to_quote` 是所有 quote 的**唯一出口**（无论缓存命中 `:97` 还是新抓取 `:103`），在此归一化可保证缓存与返回值一致。

**改法**: **保留 `_dict_to_quote` 的归一化（单一真相源），删除 fetcher 内的归一化调用**。
- `tencent_quote.py:36-37`：删除 `rec["volume"] = normalize_volume(...)` / `rec["amount"] = normalize_amount(...)`，并移除 `from common import normalize_volume, normalize_amount` 导入。
- `sina_quote.py:38-39`：同上删除（sina 源 normalize 是 no-op，但保持一致性，删除避免双层职责混淆）。
- `eastmoney_quote.py:64-65`：`"volume": d.get("f47", 0)` / `"amount": d.get("f48", 0)`（删 normalize 调用），移除导入。
- `scripts/data/__init__.py:230-231` **保持不变**（这是唯一归一化点）。

**缓存清理**: 现有缓存含被污染的值。在 `_dict_to_quote` 增加防御性注释说明归一化责任在此。**不主动清缓存**（TTL 会自然失效；主动清会拖慢首次请求）。

**新增测试**: `tests/test_data_quote.py`（或扩展现有）—— 构造 tencent 源 raw dict（volume=1000手），经 `_dict_to_quote` 后断言 `quote.volume == 100000`（×100 一次），非 `10000000`。

### A2. Tushare K线 代码映射 + 日期格式修复（P0-2）
**文件**: `scripts/fetchers/kline/tushare_kline.py`

**改法**:
- **代码映射**（:31-34）：删除 `if code.startswith(("sh","SH"))` 硬编码，改用已有的 `infer_exchange(code)`：
  ```python
  from common import BaseFetcher, plain_code, infer_exchange
  ...
  ex = infer_exchange(code)  # "sh"/"sz"/"bj"
  ts_code = f"{plain}.{ex.upper()}" if ex else f"{plain}.SZ"
  ```
  这修复 bj 代码（43/83/87/88/92 开头）被错误映射到 .SZ 的问题。
- **日期格式**（:50）：tushare `pro.daily` 返回 `trade_date` 为 `YYYYMMDD`（如 "20260108"），但系统其他地方（KlineBar.day、回测 `bars[i].day < common_start_date` 字符串比较）期望 `YYYY-MM-DD`。改为格式化：
  ```python
  td = str(row.get("trade_date", ""))
  day = f"{td[:4]}-{td[4:6]}-{td[6:8]}" if len(td) == 8 else td
  ```
  `"day": day,`

**测试**: 新增/扩展 `tests/test_fetchers.py` 或 `test_tushare_kline.py`，mock `pro.daily` 返回 `trade_date="20260108"`，断言结果 `day == "20260-01-08"`；构造 `bj430047` 断言 `ts_code == "430047.BJ"`。

### A3. 校准数据 legacy→merged 迁移（P0-3）
**文件**: `experts/calibration.py`、`data/expert_calibration.json`

**根因**: `record_prediction` 写 merged 名，`data["experts"]` 键是 legacy 名，`verify_predictions:233` 的 `in` 检查失败。

**改法**（两步）:
1. **一次性数据迁移**：在 `_load()` 内加迁移逻辑（幂等，迁移后加 `"_migrated": true` 标志避免重复）：
   ```python
   _LEGACY_TO_MERGED = {
       "buffett": "value_institution",
       "duan_yongping": "value_institution",
       "chaogu_yangjia": "emotion_tech",
       "zuoshou_xinyi": "emotion_tech",
   }
   ```
   - 对每个 legacy 键：若 merged 键已存在，合并 events/correct（相加），取较新的 last_updated；若不存在，复制并改名。删除 legacy 键。
   - `compute_calibration_factor` 已只读 active 名（:302-308），迁移后数据正好匹配。
2. **`verify_predictions` 防御**（:232）：`expert_name` 已是 active 名（record_prediction 写入），迁移后 `data["experts"]` 键也是 active 名 → `in` 检查自然通过。额外加注释说明此不变量。

**保留**: lynch/soros 是 active 名（短线），其数据不需迁移，保留原样。

**测试**: 扩展 `tests/test_calibration.py`：
- 新增 `test_legacy_migration_on_load`：构造含 buffett+duan_yongping+value_institution 的 raw 数据，`_load()` 后断言 value_institution.events == 合并值、buffett 键消失、`_migrated==true`。
- 新增 `test_verify_updates_merged_expert`：record_prediction 用 value_institution，verify_predictions 后断言 `data["experts"]["value_institution"]["events"]` 增加。
- `test_report_contains_all_experts`（:238-250）：当前断言 legacy 名在报告中。迁移后 buffett 等消失 → **改为断言 active 名**（value_institution/emotion_tech/lynch/soros/xu_xiang/zhao_laoge 等 8 个）。这是**锁定错误行为的测试**，必须改。

### A4. 回测 20cm 涨跌停检测修复（P0-4）
**文件**: `scripts/backtest/engine.py`

**根因**: `_is_limit_or_suspended(bars, idx)`（:426）签名无 `code`，内部 `getattr(bar, "code", "")`（:456）总是 `""`（KlineBar 无 code 字段），导致 `code.startswith(("sz300",...))` 永远 False → 所有股票按 10% 判涨跌停，创业板/科创板 20cm 涨跌停股被漏判。

**改法**:
- `_is_limit_or_suspended` 增加 `code: str = ""` 参数：`def _is_limit_or_suspended(bars, idx, code=""):`，删除 `getattr(bar, "code", "")` 行，直接用传入的 `code`。
- 调用处（:176）：`if _is_limit_or_suspended(bars, i, code):`（`code` 在 :182 已可用，作用域内）。
- 检查 `_is_limit_or_suspended` 的其他调用点（grep），全部传 `code`。

**测试**: 扩展 `tests/test_backtest.py`：构造 sz300 创业板 bars，某日涨幅 18%，断言 `_is_limit_or_suspended(bars, idx, "sz300xxx")` 返回 True（limit=0.195）；同数据传 `"sh600xxx"` 返回 False（未达 0.095... 实际 18%>9.5% 也 True，需构造 9.5%-18% 区间数据精确区分）。

### A5. 中枢合并交集修复（P0-5）
**文件**: `scripts/chan/zhongshu.py:50-58`

**改法**: 合并步骤从并集改为交集，与初始构建（:25-26）一致：
```python
# 交集：取较低的高点、较高的低点
new_zg = min(last["zg"], zs["zg"])
new_zd = max(last["zd"], zs["zd"])
```
更新 `mid`/`width` 计算用 `new_zg`/`new_zd`。若 `new_zd >= new_zg`（交集为空），说明合并后无意义，保留 `last` 不合并（边界处理：实际上重叠前提 `zs["zd"] < last["zg"]` 已保证交集非空，但稳妥起见加判断）。

**测试**: 扩展 `tests/test_chan.py::TestChanZhongshu`：
- 新增 `test_merge_uses_intersection_not_union`：构造两个重叠中枢（zg1=11,zd1=9 / zg2=12,zd2=10），断言合并后 `zg==min(11,12)==11`、`zd==max(9,10)==10`（交集），而非 `zg==12, zd==9`（并集）。

### A6. 双针探底死代码修复（P0-6）
**文件**: `scripts/strategies/patterns/shuangzhen.py:20`

**改法**: 删除 `lows[i - 5 : i + 1]` 这一无副作用表达式（lint 死代码）。`lows` 参数虽在此行未用，但函数签名 `detect_shuangzhen(records, closes, lows, volumes)` 的 `lows` 被 `_make_records` 测试传入且有语义意义（未来可能用 lows 做影线判断替代 r.get("low")）。**保留 `lows` 参数**（避免破坏调用方），仅删除死代码行。

**测试**: `tests/test_patterns_local.py` 现有 shuangzhen 测试覆盖正常路径，无需新增。

## 阶段 B：高影响 P1 修复（3 项）

### B1. ST 检测统一（P1）
**文件**: `scripts/data/pool.py`、`scripts/business/screening_service.py`

**根因**: `pool.py:312` 用 `"ST" in name.upper()`（子串，捕获 `*ST`/`S*ST`/中段 ST）；`screening_service.py:360` 用 `startswith(["ST","*ST"])`（漏 `S*ST` 和中段）。两套逻辑可漂移。

**改法**:
- 以 `pool.is_st`（子串匹配，更宽松、更安全）为单一真相源。
- `screening_service.py:360` 改为 `from data.pool import is_st; is_st = is_st(name)`，删除 `_st_prefixes()`/`startswith` 逻辑。保留 `limits.yaml` 的 `st_prefixes` 配置项定义但标注 deprecated（或让 `is_st` 接受可选 prefix 参数——简单起见，直接用子串）。
- **决策：子串匹配更安全**（宁可多过滤不可漏过滤 ST），与 pool 层一致。

**测试**: 扩展 `tests/test_data_pool.py`：新增 `test_is_st_catches_s_star_st`：`is_st("S*ST智胜")` 返回 True（验证当前子串行为）；扩展 screening 测试断言同行为。

### B2. 文件 IO 原子性加固（P1，最高风险 2 处）
**文件**: `scripts/data/pool.py`、`scripts/calibration_sync.py`

**根因**: `pool.py:302,536,572` 直接 `open(FILE,"w")+json.dump` 写 `all_stocks.json`/`sector_stocks.json`（全市场/全板块，大文件，崩溃即损坏）；`calibration_sync.py:190` 直接 `write_text` 写 `expert_calibration.json`（而 `experts/calibration._save` 对同一文件已用原子写——不一致）。

**改法**: 抽取共享原子写工具。`scripts/portfolio/_file_utils.py` 的 `raw_write` 已存在但属于 portfolio 子包（import 路径较深）。在 `scripts/common/utils.py` 新增轻量 `atomic_write_json(path, obj)`（tempfile.mkstemp + json.dump + os.replace + try/finally 清理），供 pool 和 calibration_sync 使用。
- `pool.py:302,536,572`：替换为 `atomic_write_json(FILE, data)`。
- `calibration_sync.py:190`：替换为 `atomic_write_json(_CALIBRATION_FILE, data)`，或直接 `from experts.calibration import _save`（但 _save 签名接受整个 data dict，需对齐——用 atomic_write_json 更解耦）。
- **范围控制**: 仅修这 3 个高频/大文件点，不强求全仓库统一（snapshots/metrics 等低风险点记入 roadmap）。

**测试**: `tests/test_common_utils.py` 新增 `test_atomic_write_json`：写文件后内容正确；mock `os.replace` 抛异常，断言临时文件被清理、原文件未损坏。

### B3. calibration_sync pull 数据丢失防护（P1）
**文件**: `scripts/calibration_sync.py`

**根因**: gist `pull()`（:190）直接 `write_text` 覆盖本地文件。若 gist 内容为空/损坏（网络问题、误操作），本地校准数据被清空。

**改法**: `pull()` 在写入前校验 gist 内容：
- 解析后必须是 dict 且含 `"experts"` 键（非空）或 `"predictions"` 键。
- 若 gist 为空 `{}` 或缺 `"experts"`，**拒绝覆盖**，记 warning 并保留本地数据。
- 与 B2 的原子写结合：先写临时文件再 replace，即使校验通过也不怕写一半崩溃。

**测试**: 扩展 `tests/test_calibration_sync.py`：mock `_fetch_gist` 返回 `{}`，断言 `pull()` 不覆盖本地、返回失败状态。

- **A3** 数据迁移不可逆（删除 legacy 键）。但 legacy 专家已废弃，数据合并到 merged 后辈是正确语义。git 可回滚 json 文件。
- B 系列改动局部、低风险。
